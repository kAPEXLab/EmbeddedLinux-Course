User Programs Access Devices: Syscalls  
(1/3)  
Trace how a user program interacts with a device via syscalls → VFS → driver.  
1. Trace syscalls with strace  
$ cd ~  
$ mkdir demo && cd demo  
$ strace -o trace.txt -f cat /dev/zero | head -c 1024 > /tmp/junk  
$ sed -n '1,60p' trace.txt  
What this does:  
• cat /dev/zero → produces an infinite stream of zero bytes.  
• head -c 1024 → consumes only the first 1024 bytes, then exits.  
• strace -o trace.txt -f → records syscalls from cat (and its children if any; head isn’t a child here, so it won’t be  
traced).  
• Output redirected to /tmp/junk.

(2/3)  
Terminal will show details like this:  
407636 execve("/usr/bin/cat", ["cat", "/dev/zero"], ...) = 0 → User-space program starts.  
407636 openat(AT_FDCWD, "/dev/zero", O_RDONLY) = 3 → User space requests open(); kernel VFS routes to /dev/zero driver.  
407636 read(3, "\177ELF\2\1\1\3...", 832) = 832 407636 read(3, "\0\0\0\0\0\0\0...", 131072) = 131072 → User space calls read(); kernel  
returns bytes. For /dev/zero, it *generates zeros*.  
407636 write(1, "\0\0\0\0\0\0\0...", 131072) = 65536 → User space calls write(); kernel writes to a pipe (stdout → head).  
407636 --- SIGPIPE {si_signo=SIGPIPE, ...} ---  
407636 +++ killed by SIGPIPE +++ → head consumed 1024 bytes and closed the pipe; kernel signals cat with SIGPIPE.  
Key mapping:  
• User space: cat issues open, read, write calls.  
• Kernel space: VFS + device driver /dev/zero + pipe  
implementation handle the actual work.  
• write() returning 65536 (not 131072) is due to pipe  
capacity.  
cat --read --> /dev/zero driver  
--write --> pipe buffer  
head reads 1024 bytes → closes → SIGPIPE to cat

Why /dev/zero ≠ Regular File?  
Why does /dev/zero behave differently from a regular file?  
• /dev/zero is a character device exposed via a special device node.  
• /dev/zero is part of major number 1 → memory devices  
• It’s backed by a kernel driver that returns zero bytes on read indefinitely.  
• No disk access occurs; it’s generated in kernel space.  
A regular file has finite size stored on disk; /dev/zero is infinite and virtual.

• Each directory represents a device class exposed by sysfs.

• net → Network interfaces.  
• thermal → Thermal sensors.  
• tty → Terminals.  
• power_supply → Battery/AC power info.

Explanation:  
• enp0s31f6 → Wired Ethernet interface.

• virbr0 → Virtual bridge (used by virtualization).  
• wlp0s20f3 → Wireless interface.  
• zcctun0 → Likely a VPN or tunnel interface.

Device Nodes & Major/Minor Numbers  
ls -l /dev/null /dev/zero /dev/tty  
crw-rw-rw- 1 root root 1, 3 Jan 15 11:20 /dev/null  
crw-rw-rw- 1 root tty 5, 0 Jan 15 11:20 /dev/tty  
crw-rw-rw- 1 root root 1, 5 Jan 15 11:20 /dev/zero  
• Leading c → Character device (stream-oriented, e.g., /dev/null, /dev/zero, /dev/tty).  
• If it were b, it would be a block device (disk-like, e.g., /dev/sda).  
• Permissions (rw-rw-rw-) → Anyone can read/write these devices.  
• Owner/Group (root root) → Owned by root.  
• 1, 3 or 1, 5 → Major, Minor numbers:  
• Major = Identifies the driver in the kernel (e.g., 1 = memory devices).  
• Minor = Identifies the specific device handled by that driver (3 = /dev/null, 5 = /dev/zero).  
• Path → /dev/null, /dev/zero, /dev/tty are device nodes, not regular files.  
What happens if you delete /dev/null?  
• You lose the handle to the driver;  
• programs that rely on /dev/null (for discarding output) will fail.  
• The driver still exists in the kernel, but the user-space entry point is gone.

“Why do we see multiple entries for /usr/bin/sleep?”  
• Expected answer: Different sections (code, data) with different permissions.  
“What does [heap] mean?”  
• Expected answer: Dynamic memory region for allocations.  
“Why is libc.so.6 mapped?”  
• Expected answer: Because sleep uses libc functions.
